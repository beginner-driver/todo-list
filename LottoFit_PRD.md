# 로또핏 (LottoFit) — 제품 요구사항 정의서 (PRD) v2.1

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| **제품명** | 로또핏 (LottoFit) |
| **한 줄 정의** | 동행복권 역대 당첨 데이터의 통계(홀짝 균형, 총합, 최소/최대값)를 분석하고, 이를 만족하는 황금 구간 조합만 엄선하여 제공하는 스마트 로또 추천 웹 서비스 |
| **기획 배경** | 매 회차 추첨은 독립시행이지만, 실제 당첨 이력이 증명하는 '통계적 메이저 영역(황금 구간)' 안에서 번호를 선별하려는 유저 수요가 많음. 이를 과학적이고 신뢰감 있는 UI/UX로 시각화하여 재미와 심리적 안정감을 극대화 |

> **⚠️ 법적 디스클레이머 (필수 UI 노출)**
> "본 서비스는 역대 당첨 통계 기반 참고용 번호 조합을 제공하며, 당첨 확률을 높이거나 보장하지 않습니다. 로또 추첨은 매 회차 독립적인 확률 시행입니다."

## 2. 핵심 알고리즘 파이프라인

```
[ 1단계: 무작위 추출 ] ──► [ 2단계: 최소/최대 필터 ] ──► [ 3단계: 홀짝 균형 필터 ] ──► [ 4단계: 총합 범위 필터 ] ──► [ 최종 반환 ]
      (필수)                    (토글 가능)                   (토글 가능)                    (토글 가능)
```

### 2.1 각 단계 정의

| 단계 | 역할 | 임계값 산출 방식 | 기본값 |
|------|------|------------------|--------|
| **1단계** 무작위 추출 | 1~45에서 6개 번호를 무작위 추출 | — | 항상 실행 (끌 수 없음) |
| **2단계** 최소/최대 필터 | 번호 6개의 **분포 범위** 검증. 시작수(최소)와 끝수(최대)가 역대 관측 분포 구간 안에 있는지 확인 | 역대 회차별 시작수/끝수 분포에서 상위 80% 커버 구간을 산출 | ON |
| **3단계** 홀짝 균형 필터 | 번호 6개의 **홀짝 균형도** 검증. 극단적 비율(6:0, 0:6, 5:1, 1:5)을 걸러내고 역대 출현 빈도가 높은 균형 조합만 통과 | 역대 회차별 홀짝 비율 조합(7종) 빈도를 집계, 출현 빈도 상위 조합(예: 3:3, 4:2, 2:4)만 허용 | ON |
| **4단계** 총합 범위 필터 | 번호 6개 **합계가 역대 정규분포 구간 안에 위치**하는지 검증 | 역대 회차별 총합의 평균과 표준편차 산출, 평균 ±1σ 구간을 황금 구간으로 적용 | ON |

### 2.2 임계값 정책

- 모든 임계값은 **하드코딩 금지** — DB에 축적된 역대 당첨 데이터로부터 통계적으로 산출
- 신규 회차 반영 시(매주 배치) 임계값을 함께 **재계산하여 갱신**
- 필터 통과 구간은 역대 분포의 상위 60~80%를 커버하도록 설정하여, "통계적 신뢰감"과 "다양한 조합이 나오는 재미"를 양립

### 2.3 사용자 필터 토글

- 2~4단계는 사용자가 UI에서 **개별 ON/OFF 가능**
- **최소 1개 이상 활성화 필수** — 전체 비활성화 시도 시 "순수 랜덤 모드가 됩니다" 경고 문구 노출 후 차단
- 비활성화된 필터는 스펙 리포트에서 "미적용" 표시

### 2.4 재시도 정책

- 필터 조건을 모두 만족하는 조합이 나올 때까지 1단계부터 재추출
- **재시도 상한: 최대 1,000회** — 상한 초과 시 그 시점까지 가장 많은 필터를 통과한 차선 조합을 반환하고, "일부 필터 조건을 완화하여 추천되었습니다" 안내 노출
- 사용자 설정(필터 ON 수, 게임 수)에 따라 재시도 빈도가 달라지므로 프론트엔드 로딩 시간도 탄력적으로 대응

## 3. 데이터 파이프라인

### 3.1 데이터 수집 방식

동행복권은 **공식 오픈API를 제공하지 않음**. 아래 비공식 JSON 엔드포인트를 활용하여 공개 추첨 결과를 조회한다.

```
GET https://www.dhlottery.co.kr/common.do?method=getLottoNumber&drwNo={회차번호}
```

**응답 예시 (drwNo=1121)**
```json
{
  "returnValue": "success",
  "drwNo": 1121,
  "drwNoDate": "2024-05-25",
  "drwtNo1": 3, "drwtNo2": 11, "drwtNo3": 18,
  "drwtNo4": 24, "drwtNo5": 31, "drwtNo6": 42,
  "bnusNo": 7,
  "totSellamnt": 112543500000,
  "firstWinamnt": 2345678900
}
```

- 존재하지 않는 미래 회차 요청 시 `"returnValue": "fail"` 반환 → 이를 종료 조건으로 최신 회차 자동 감지

### 3.2 수집 정책 (v2.1: 자동 수집 없음, 수동 갱신)

| 구분 | 정책 |
|------|------|
| **최초 백필** | `drwNo=1`부터 순차 요청, 요청 간 **최소 1~2초 딜레이**로 직접 1회 수집 |
| **갱신 방식** | 서버 크론 없이, 매주 추첨 후 **관리자가 직접** `data/draws.json`에 1건 수동 추가 → `git push` |
| **요청 간격** | 순차 요청 시 최소 1초 간격 유지 |
| **User-Agent** | 일반 브라우저 값 사용 |
| **실패 재시도** | 지수 백오프(1초→2초→4초), 최대 3회. 3회 실패 시 로그 기록 후 다음 회차로 진행 |
| **리스크 인지** | 비공식 엔드포인트로 향후 차단/변경 가능성 존재 |

### 3.3 수동 트리거

- **일반 사용자 화면에 "DB 최신화" 버튼 없음** — 보안 위험(DoS 유발), 불필요(추첨일 외엔 새 데이터 없음)
- 데이터 갱신은 저장소 내 `data/draws.json` 파일을 직접 수정 → push하는 방식으로만 수행
- 프론트엔드에는 **읽기 전용 상태 표시**만 노출: "최근 반영 회차: 1121회 (2024.05.25)"

## 4. 기능 요구사항

### 4.1 수량 선택

| 구분 | 상세 |
|------|------|
| **퀵 버튼** | 1게임, 2게임, 3게임 (가장 선호하는 수량을 원터치 선택) |
| **직접 입력** | "직접 입력" 버튼 클릭 시 숫자 Input 필드 동적 노출. **최소 1게임, 최대 10게임** 유효성 검사 적용 |
| **필터 토글** | 2~4단계 필터별 ON/OFF 토글 스위치 (기본값: 전체 ON, 최소 1개 필수) |

### 4.2 예측 번호 스펙 리포트

추천된 각 게임을 클릭/마우스 오버 시 아래 통계 정보를 차트 + 텍스트로 제공:

| 항목 | 시각화 방식 | 비고 |
|------|-------------|------|
| **총합 위치** | 정규분포 곡선 상 핀 포인트 표시 | "전체 회차 중 상위 N% 위치" 텍스트 병기 |
| **시작수/끝수 구간** | 확률 지표 텍스트 | 해당 버킷의 역대 출현율 표시 |
| **홀짝 비율** | 시각적 막대 그래프 | 홀 N개 / 짝 N개 표시 |
| **적용 필터 상태** | 태그/뱃지 | 비활성화 필터는 "미적용" 표시 |

### 4.3 부가 편의성 기능

| 기능 | 상세 |
|------|------|
| **번호 저장 (하트)** | 하트 아이콘 클릭 → 통통 튀는 모션 → 보관함에 저장 |
| **이미지 저장** | 영수증 카드 UI를 이미지로 디바이스에 저장 |
| **공유** | 카카오톡 공유 (카카오 SDK 연동 필요) |

### 4.4 보관함 저장 방식

- **v1: 익명 로컬 저장** — 브라우저 LocalStorage + 고유 키(`anon_xxxxx`) 기반, 계정 시스템 없음
- 추후 로그인 기반 클라우드 동기화로 확장 가능 (v2 이후 검토)

## 6. 기술 스택

> **v2.1 변경**: 별도 서버(Vercel/SQLite) 없이 **GitHub 생태계만으로 운영**하는 구조로 전환. 프론트는 GitHub Pages, 백엔드는 Cloudflare Workers(서버리스), DB는 저장소 내 JSON 파일로 대체.

### 6.1 프론트엔드

| 우선순위 | 프레임워크 | 장점 | 단점 |
|----------|-----------|------|------|
| **1 (채택)** | **정적 React (Vite) + Tailwind CSS, GitHub Pages 배포** | 빌드 결과물이 순수 정적 파일이라 GitHub Pages와 궁합 최상, `git push` 시 Actions로 자동 배포 | SSR/SEO 없음 (단, 이 서비스는 검색 유입보다 재방문 중심이라 큰 손실 아님) |
| 2 | Next.js (Static Export) + Tailwind | React 생태계 그대로 사용, `next export`로 정적 배포 가능 | API Routes 등 서버 기능은 GitHub Pages에서 무의미 — 쓰지 않을 기능 포함 |
| 3 | SvelteKit (Static Adapter) + Tailwind | 빌드 결과물 가장 가벼움, 애니메이션 내장 transition | 생태계/유지보수 인력 확보 어려움 |

### 6.2 백엔드

| 우선순위 | 방식 | 장점 | 단점 |
|----------|------|------|------|
| **1 (채택)** | **Cloudflare Workers** | 서버 관리 불필요(서버리스), 무료 한도로 충분(1일 10만 요청), 프론트와 동일한 JS/TS, `wrangler deploy` 한 줄로 배포 | 프론트(`github.io`)와 도메인이 달라 **CORS 설정 필수**, 배포가 프론트와 별도 파이프라인 |
| 2 | Vercel API Routes | 설정 없이 Next.js와 완전 통합 | GitHub Pages를 프론트로 쓰는 전제와 맞지 않음 (구조 변경 필요) |
| 3 | 백엔드 없음 (브라우저 JS로 전량 계산) | 배포/CORS 이슈 자체가 없음, 가장 단순 | "백엔드 유지" 요구사항과 배치됨 |

> **채택 근거**: 서버 운영 없이 백엔드 로직(필터링/재추출/스펙 리포트 계산)을 유지하려면 서버리스가 유일한 선택지. Cloudflare Workers는 무료로 충분하고 별도 인프라 지식 없이 배포 가능.

### 6.3 데이터베이스

| 항목 | 선택 | 근거 |
|------|------|------|
| **DB** | **JSON 파일 (`draws.json`), 저장소 내 커밋** | 자동 수집 없이 **수동으로 회차 데이터를 갱신**하는 정책(3.1 참고)과 일치. 별도 DB 서버 불필요, Workers가 파일을 읽어 계산 |
| **갱신 방식** | 매주 추첨 후 `draws.json`에 1건 수동 추가 → `git push` → Workers가 즉시 반영 | 서버 크론 없이도 데이터 최신화 가능 |
| **확장 경로** | 데이터량이 커지거나 자동 수집이 필요해지면 실제 DB(SQLite/PostgreSQL) + 서버로 재전환 검토 | 현재 규모(약 1,200행)는 JSON으로 충분 |

### 6.4 저장소 구조

```
LottoFit/
├── frontend/     ← GitHub Pages로 배포 (정적 빌드 결과물)
├── worker/       ← Cloudflare Workers 소스 (wrangler deploy)
└── data/
    └── draws.json  ← 역대 당첨 데이터 (수동 갱신)
```

### 6.5 주요 의존성

| 용도 | 패키지/서비스 |
|------|---------------|
| 애니메이션 | Framer Motion (또는 CSS transition) |
| 이미지 저장 | html2canvas 또는 dom-to-image |
| 카카오 공유 | Kakao JavaScript SDK |
| 프론트 배포 | GitHub Actions → GitHub Pages |
| 백엔드 배포 | `wrangler deploy` (Cloudflare Workers) |
| CORS | Workers 응답 헤더에 `Access-Control-Allow-Origin: https://beginner-driver.github.io` 명시 |

## 7. DB 스키마

### 7.1 draws (역대 당첨 데이터)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| draw_no | INTEGER (PK) | 회차 번호 |
| draw_date | TEXT | 추첨일 (YYYY-MM-DD) |
| num1~num6 | INTEGER | 당첨번호 6개 (오름차순) |
| bonus_no | INTEGER | 보너스 번호 |
| total_sales | INTEGER | 총 판매금액 |
| first_prize | INTEGER | 1등 당첨금 |

### 7.2 filter_thresholds (산출 임계값 캐시)

| 컬럼 | 타입 | 설명 |
|------|------|------|
| filter_type | TEXT (PK) | `sum_range` / `odd_even` / `min_max` |
| params | TEXT (JSON) | 산출된 임계값 JSON |
| calculated_at | TEXT | 최종 산출 시각 |
| based_on_draws | INTEGER | 산출 기준 회차 수 |

### 7.3 saved_games (보관함 — 프론트 LocalStorage 구조)

```json
{
  "saved_games": [
    {
      "id": "sg_001",
      "numbers": [4, 15, 23, 27, 33, 41],
      "saved_at": "2026-07-16T20:46:10+09:00",
      "filters_applied": ["min_max", "odd_even", "sum_range"]
    }
  ]
}
```

## 8. API 목업 데이터

### 8.1 당첨 데이터 원본 (DB 저장용)

```json
{
  "draw_no": 1121,
  "draw_date": "2024-05-25",
  "numbers": [3, 11, 18, 24, 31, 42],
  "bonus_no": 7,
  "total_sales": 112543500000,
  "first_prize": 2345678900
}
```

### 8.2 추천 요청 (프론트 → 백엔드)

```json
{
  "game_count": 3,
  "filters": {
    "min_max": true,
    "odd_even": true,
    "sum_range": true
  }
}
```

### 8.3 추천 결과 응답 (백엔드 → 프론트)

```json
{
```

## 9. 비기능 요구사항

| 항목 | 요구사항 |
|------|----------|
| **트래픽 대응** | 토요일 20:45 추첨 직후 스파이크 예상 — Cloudflare Workers 무료 한도(1일 10만 요청) 내에서 충분히 대응 |
| **응답 속도** | 추천 API 응답 2초 이내 (로딩 연출 1.5초 포함) |
| **보안** | 게임 수 입력 유효성 검사(1~10), 필터 최소 1개 필수, Workers CORS 허용 origin 고정 |
| **접근성** | 번호 색상 외 숫자 텍스트 병기 (색각 이상 대응) |
| **반응형** | 모바일 퍼스트 — 주 사용 환경이 스마트폰 |
